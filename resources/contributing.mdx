## Introduction

Thank you for considering contributing to Laravel Active Email! Contributions from the community help make this package better for everyone.

<Info>
All contributions, whether bug reports, feature suggestions, documentation improvements, or code changes, are welcome and appreciated.
</Info>

## Ways to Contribute

There are several ways you can contribute to this project:

- Report bugs and issues
- Suggest new features or enhancements
- Improve documentation
- Submit pull requests with code improvements
- Add new disposable email domains to the blacklist
- Write or improve tests

## Reporting Issues

If you discover a bug or have a feature request, please open an issue on GitHub:

1. Visit the [Issues page](https://github.com/veeqtoh/laravel-active-email/issues)
2. Check if the issue already exists
3. If not, create a new issue with a clear title and description
4. Include:
   - Steps to reproduce the issue (for bugs)
   - Expected vs actual behavior
   - Your environment (PHP version, Laravel version, package version)
   - Any relevant code samples or error messages

<Note>
The more information you provide, the easier it will be to diagnose and fix the issue.
</Note>

## Security Issues

If you discover a security vulnerability, please **do not** open a public issue.

<Warning>
**Security vulnerabilities should be reported privately** to protect users while a fix is being developed.
</Warning>

Instead, please email the security issue directly to:

**Victor Ukam**: [victorjohnukam@gmail.com](mailto:victorjohnukam@gmail.com)

You will receive a response within 48 hours acknowledging your report. We'll work with you to understand the issue and develop a fix.

## Submitting Pull Requests

We welcome pull requests for bug fixes, new features, and improvements!

### Before You Start

1. **Fork the repository** on GitHub
2. **Clone your fork** locally
3. **Create a new branch** for your feature or fix:
   ```bash
   git checkout -b feature/your-feature-name
   ```
   or
   ```bash
   git checkout -b fix/your-bug-fix
   ```

### Development Guidelines

1. **Follow PSR-12 coding standards** for PHP code
2. **Write tests** for new functionality
3. **Update documentation** if you're changing functionality
4. **Keep commits focused** - one feature/fix per pull request
5. **Write clear commit messages** that describe what changed and why

### Code Quality

Before submitting your pull request:

1. **Run the test suite** to ensure all tests pass:
   ```bash
   vendor/bin/pest
   ```

2. **Ensure your code follows the project style**
3. **Check for any linting errors** if using PHPStan/Larastan
4. **Update or add tests** for your changes

### Submitting Your PR

1. **Push your branch** to your fork on GitHub
2. **Open a pull request** against the `main` branch of the original repository
3. **Provide a clear description** of:
   - What the PR does
   - Why the change is needed
   - How to test it
   - Any breaking changes
4. **Reference related issues** using GitHub's linking syntax (e.g., "Fixes #123")

### After Submission

- Be responsive to feedback and questions
- Make requested changes in new commits (don't force-push)
- Once approved, your PR will be merged

## Adding Disposable Domains

One of the most valuable contributions you can make is adding newly discovered disposable email domains to the blacklist.

If you find a disposable email service that isn't currently blocked:

1. Add the domain to the appropriate list in the package's domain files
2. Include evidence or documentation about the service
3. Submit a pull request with your addition
4. Explain how you discovered the domain (if relevant)

## Testing

This package uses [Pest](https://pestphp.com/) for testing. Tests are located in the `tests/` directory.

### Running Tests

Run the full test suite:
```bash
vendor/bin/pest
```

Run specific test files:
```bash
vendor/bin/pest tests/Feature/ValidationTest.php
```

Run with coverage:
```bash
vendor/bin/pest --coverage
```

### Writing Tests

When adding new functionality:

1. Write tests that cover the new behavior
2. Ensure existing tests still pass
3. Test both success and failure cases
4. Include edge cases and boundary conditions

Example test structure:
```php
test('it blocks disposable email addresses', function () {
    $validator = validator(['email' => 'test@mailinator.com'], [
        'email' => ['email', 'notblacklisted'],
    ]);
    
    expect($validator->fails())->toBeTrue();
});
```
